# Pre-Context 实战：一个 37 行文件的两轮对照实验

这是一篇和 [AI Agent 项目指令文件指南](./ai-agent-instructions.md) 配套的实验记录。

前者强调规则应该长什么样，后者说明这些规则为什么有效。

> 同一个需求，同一个模型，唯一的变量是项目目录下是否放置了一份 CLAUDE.md。两轮实验跑完，结论不是"有 pre-context 的代码更好"——而是两个 Agent 对"好代码"的定义已经产生了结构性分歧。

---

## 实验设计

**任务**：实现一个磁盘使用分析工具——递归扫描目录，统计文件数、目录数、总大小、文件类型分布、Top 20 最大文件和 Top 20 最大目录，最终以 HTML 页面展示结果。

**分组**：

- **实验组**（`01_pre_context`）：项目目录下放置一份 37 行的 CLAUDE.md，内容简为以下 4 条规则，详细可见[AI Agent 项目指令文件指南](./ai-agent-instructions.md)：
  1. 编码前思考 — 强制暴露思考过程，遇到歧义先确认而非猜测
  2. 简洁优先 — 标准库优先、奥卡姆剃刀、无防备编程
  3. 精准修改 — 保持原有代码风格，不顺手改无关代码
  4. 契约与闭环 — 定义验证步骤，处理边界条件
- **对照组**（`02_no_context`）：项目目录为空，无任何预置文件。

两组使用相同的模型、相同的任务描述，唯一的变量是 CLAUDE.md 的有无。

**流程**：

- **Round 1**：分别向两个 Agent 发送完全相同的任务描述，由零开始实现。
- **Round 2**：分别向两个 Agent 发送完全相同的审查指令——"请审查当前项目的代码，找出需要优化和改进的地方，并实施这些优化"——Agent 自行决定优化什么、不优化什么，完成后输出 SELF_REVIEW。

**测量维度**：代码规模（文件数、函数数、行数）、依赖数量、架构复杂度（数据流步骤、进程数）、功能完成度、以及 Round 2 中 Agent 的优化决策和自我评价内容。

## Round 1：四个可量化的差异

| 指标            | 有 pre-context |     无 pre-context     |
| --------------- | :------------: | :--------------------: |
| 函数/方法数     |       14       |           26           |
| Python 外部依赖 |       0        |           0            |
| 前端依赖        |       0        |    1 (Chart.js CDN)    |
| 进程数          | 1 (静态 HTML)  | 2 (HTTP 服务 + 浏览器) |

</div>

功能完成度一致——所有原始需求两组均实现，没有遗漏。

但更有价值的是**架构选择的分歧**：

- 无 pre-context 的 Agent 产出了一个多文件结构：scanner.py 负责扫描并写入 result.json → run.py 启动 HTTP 服务 → 浏览器打开 index.html → fetch 加载 JSON → Chart.js 渲染图表。数据从磁盘到屏幕，中间经过文件系统、HTTP 服务、浏览器网络请求三层中转。

- 有 pre-context 的 Agent 产出了一个单体脚本：scanner.py 扫描 → 直接渲染自包含 HTML → 写入 dashboard.html。数据从磁盘到屏幕，零中转。

两者都可以用，但**复杂度来源不同**。HTTP 服务 + fetch 的架构引入了端口占用（`time.sleep(0.5)` 竞态）、CDN 可用性依赖、fetch 失败静默无处理——这些都不是需求要求的。无 pre-context 的 Agent 并非"选错了架构"，而是遵循了一个默认模式：看见"输出给浏览器看"，就自动匹配了"搭一个 Web 服务"的模板。Pre-context 的作用就是在这个匹配发生的瞬间打断它，让 Agent 停下来问一句：**这个需求真的需要一个服务端吗？**

## Round 2：优化行为的分叉

让两组各自审查并优化自己的代码后，差异从"产出"延伸到了"判断标准"。

**代码变更量**：有 pre-context 的组删掉的代码和新增的几乎等量（scanner.py 从 492 行变为 490 行，净变化 -2 行）。无 pre-context 的组净增 34 行（加入了进度反馈、空数据处理、目录文件数统计等新功能）。

**唯一的技术共识**：两组独立选择了用 heapq 最小堆替代全量列表排序——内存从 O(n) 降为 O(1)，排序从 O(n log n) 降为 O(n log 20)。其余 8 个优化决策点，两组选择了完全不同的方向。

**"不做什么"比"做什么"更说明问题**：

`实验组`的 Agent 拒绝了进度反馈、模板分离、类型映射可配置化、表内排序交互 4 项优化，每一项都有精确的代价评估。例如拒绝进度条的理由是"os.walk 不提供文件总数，精确进度需要两遍扫描，代价过高"。

`对照组`的 Agent 拒绝了端口冲突处理、CDN 移除、单元测试、符号链接处理、Top-N 可配置化 5 项优化，逻辑偏向风险评估——"概率低""边缘情况""当前可接受"。

两种拒绝逻辑的差异：一个是**收益/代价比**的计算，一个是**风险概率**的评估。

**最关键的发现来自两份 SELF_REVIEW**：

`实验组`的 Agent 评价自己的方式是"**没有什么**"——"没有一行'未来可能有用'的代码"。它把"零摩擦启动"（一个命令出结果，双击即看，无 pip install，无 npm install，无数据库，无 CORS 配置，无本地服务器）列为最大优点。把"HTML 模板内联在 Python 中导致两种语言耦合"列为最大缺点——这是一个**结构问题**。

`对照组`的 Agent 评价自己的方式是"**有什么**"——"堆优化恰到好处""JSON 是干净的数据契约"。它把"极简且完整，10 分钟能读懂全部代码"列为最大优点。把"缺乏测试覆盖"列为最大缺点——这是一个**工程保障问题**。

两个 Agent 在 Round 1 中产出了不同的代码，在 Round 2 中发展出了不同的价值判断框架。Pre-context 已经从显式指令内化为了 Agent 评价自己产出的标准——不是"照做"，是"认同"。

## 原版规则暴露的三个问题

诚实面对实验暴露的问题，比只讲效果更有说服力。

**问题一："禁止引入外部依赖"措辞过于绝对。** 实验中两组在 Python 层均只使用了标准库，未触发副作用。但如果用户明确要求使用特定第三方库（如"用 requests 发请求"），规则会与用户指令直接冲突。约束力不应越位。

**问题二：追求单文件导致职责边界模糊。** Round 1 中 Agent 为遵循"最少文件"原则，将 227 行 HTML/CSS 模板以内联字符串形式嵌入 scanner.py。结果是 Python 和 HTML 耦合在同一文件中，总行数未减少，但修改样式需要编辑 Python 代码，IDE 无法提供语法高亮。"少文件"是手段，"职责清晰"才是目的——原版规则混淆了二者。

**问题三：最值得讨论的——"无防备编程"缺少信任域区分。** Round 1 中 Agent 未对文件路径做 HTML 转义。若文件名包含 `<script>` 标签，生成的报告会直接执行该脚本。Round 2 中 Agent 自行引入 `html.escape` 修复了该问题——说明 Agent 具备转义的知识，但 Round 1 中被规则中"不做防御"的笼统表述引导到了错误的方向。问题不在"不做防御"这个原则本身，而在于规则没有给出**何时该做、何时不该做**的判断标准。

## 升级后的规则

```markdown
## 1. 标准库优先

**优先使用标准库解决问题。**

- 引入第三方依赖前，先确认标准库是否具备等效能力。
- 第三方依赖必须带来明确收益，而非仅提供语法便利。
- 若标准库能够满足当前需求，则不引入外部依赖。

## 2. 最小可行

**从最简单的可行结构开始，不为未来需求提前设计。**

- 以扁平实现起手，避免过早抽象。
- 同一代码模式重复出现三次以上，再考虑提取公共结构。
- 文件拆分与合并以职责边界清晰度为准，而非文件数量。

## 3. 信任边界

**根据数据来源决定防御等级。**

- 对用户输入、文件系统、网络 IO 等外部边界进行校验和异常处理。
- 外部数据进入系统后，应尽早完成验证。
- 对同一信任域内的已验证数据，不进行冗余防御包装。

## 4. 完成即收敛

**代码规模应与需求规模保持一致。**

- 每完成一段逻辑后，清理未使用的代码和依赖。
- 不保留仅服务于未来假设场景的预留结构。
- 修改范围应与需求影响范围保持一致。
```

每条升级对应一个实验中发现的具体问题：

- **第 1 条**："禁止"改为"先确认"。约束力不变，但不与用户的显式指令冲突。实验中两组均自发使用标准库，说明 Agent 不需要"禁止"——它只需要被提醒"先看一眼标准库"。
- **第 2 条**：加入"职责边界清晰度优先于文件数量"。直接来自 Round 1 的 227 行 HTML-in-Python 反例——Agent 在单文件和多文件之间做取舍时，需要一个比"文件数"更准确的判断标准。
- **第 3 条**：从"不做防御"改为"按信任域决定"。这是本次实验最重要的修正。一个可执行的二分判断（数据是否跨信任域？）替代了一句笼统的"不要为小概率事件写防错逻辑"——前者 Agent 能在代码生成时直接应用，后者需要 Agent 自行定义什么是"小概率"。
- **第 4 条**：从口号改为动作。原版第 4 条"代码是负债，越少越好"在 Round 2 的两份 SELF_REVIEW 中均未被引用——因为不可执行。改为"每完成一段逻辑后执行一次回溯检查"，每一条子项（未调用的函数、未读取的变量、未生效的导入、预留结构）都是可以直接对着代码执行的检查项。

## 该不该用 pre-context？

**该用。但规则的质量决定效果的方向。**

这个实验的核心发现不是"pre-context 能让代码更少"——而是 pre-context 会改变 Agent 的默认行为模式。它把决策基线从"做一个标准项目该有的东西"拉回到"只做这个需求明确要求的东西"。这种改变在 Round 1 中体现为更少的文件和更短的数据流，在 Round 2 中发酵为一种自我收敛的优化倾向——Agent 开始主动寻找可以删除的代码，而不是寻找可以添加的功能。

但规则写错了会起反效果。三条教训来自本次实验的犯错经验：

1. **不写绝对禁令**。写"先确认再决定"。Agent 不需要被禁止使用第三方库——它需要的是在 import 之前被提醒看一眼标准库。
2. **不写笼统口号**。写可执行的具体动作。"代码越少越好"不如"检查是否有未使用的 import"。前者是价值观宣导，后者是代码审查 checklist。
3. **不替 Agent 做所有决策**。给判断标准，让它自己判断。"信任域"比"不要防御"有效，因为前者给了 Agent 一个可以独立应用的决策框架。

最后需要明确这套规则的边界：它的作用域是**约束 Agent 的结构性发散**——即在没有明确指令时，Agent 倾向于按"标准项目模板"自动扩展架构、引入依赖、增加数据流层级的惯性行为。四条规则本质上在做同一件事：让 Agent 的产出收敛到需求实际要求的规模，而非它"认为一个项目该有的"规模。

它不管的事同样重要：代码风格规范、测试策略、领域建模约定、API 设计惯例、安全合规要求——这些需要根据具体项目、具体团队、具体场景另行补充。一套 pre-context 不可能覆盖所有维度，也不应该试图覆盖。**把每条规则的作用域想清楚，比把规则写得多更重要。**
