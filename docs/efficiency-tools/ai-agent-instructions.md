# 用项目指令文件约束 AI Agent

随着 Claude Code、Codex、Cursor 等 AI Agent 的普及，在项目根目录维护`CLAUDE.md`、`AGENTS.md`、`.cursor/rules/*`等文件以及成为一种越来越常见的实践。

这类文件通常被称为项目指令（Project Instructions）或预上下文（Pre-context）。笔者使用 Claude Code 较多，故下文统一以`CLAUDE.md`代指此类文件。

`CLAUDE.md`的作用类似于一个`repo`的`README.md`，只是区别在于前者是给 AI Agent 工具看的，后者是给开发者/用户看的。具体体现在，当 Agent 开始工作时，`CLAUDE.md`中的内容会被优先注入上下文，从而影响后续决策。   

在大量 Agent 实践中，一个越来越明显的现象是：`CLAUDE.md` 已经不仅仅是一份项目指令清单。那些被反复强调的原则会逐渐沉淀为 Agent 的决策偏好，影响其分析问题、修改代码以及进行方案权衡的方式，甚至改变它对“好代码”的判断标准。

无论是在日常 AI Coding 过程中，还是在后文的对照实验中，笔者都认同这种倾向：相比模型能力本身，持续注入的行为约束往往更能影响 Agent 最终表现出的工程风格。   

某种意义上，编写 `CLAUDE.md` 的过程，就是在训练一个更符合自己工程理念的协作者。拥有一个与开发者遵循相同开发原则的 `Copilot`，大概就是 AI 时代程序员特有的乐趣/折磨之一。

## 为什么需要预上下文

AI Agent 最大的问题并不是不会写代码，而是：

* 喜欢过度设计
* 喜欢顺手重构
* 喜欢引入依赖
* 喜欢猜测需求
* 喜欢修改无关代码

对于长期维护的项目来说，这些行为往往比实现功能本身更危险。技术栈是项目相关的，但行为约束是跨项目的。

因此需要提前告诉 Agent：

```
哪些事情应该做
哪些事情不应该做
```

以一个单一的、用于改善 Claude Code 的行为的 CLAUDE.md 文件为例，源自 Andrej Karpathy 的观察，关于 LLM 编码陷阱的总结。这个CLAUDE.md 正是为了解决上述问题而提出的。


每个项目的技术栈、目录结构和工程规范都可能不同，因此项目级规则通常需要根据实际情况定制。

相比之下，行为约束具有更强的普适性。无论使用什么语言或框架，「先理解问题」「保持实现简单」「只修改必要内容」「验证结果正确」都是长期工程实践中沉淀下来的共识。

Karpathy 的这份 CLAUDE.md 正是聚焦于这一层面。它不关心具体技术选型，而是约束 Agent 应该如何工作。


## Karpathy 风格 CLAUDE.md

```md
## 1. 编码前思考

**不要盲目开工。强制暴露思考过程。**

- **使用 `<thinking>` 标签**：在代码块之前，简要列出你的理解、假设和权衡。

- **遇到歧义即阻断**：如果需求有多种解释，停止生成代码，列出选项 A 和 B，询问用户选择。

- **评估影响半径**：思考“这个改动会影响系统中的其他模块吗？”

## 2. 简洁优先

**代码是负债，越少越好。拒绝过度推测。**

- **标准库优先**：除非用户明确要求，否则禁止引入新的第三方依赖。

- **奥卡姆剃刀**：如果不需要复杂的泛型、接口或设计模式，就写最扁平、直白的函数。

- **无防备编程**：不要为了“未来可能的需求”或“极小概率的边缘情况”编写复杂的防错逻辑，除非这涉及到核心安全性或用户已明确提出。

## 3. 精准修改

**保持原有代码的纯洁性。禁止“顺手牵羊”。**

- **严禁使用 `...` 偷懒**：在提供修改后的代码时，必须提供**完整且可运行**的代码块，或者使用清晰的 Diff 格式。绝对不要用注释如 `// ... 现有代码保持不变 ...` 来糊弄用户，这会导致复制粘贴失败。

- **风格变色龙**：即使原代码的命名规范或格式化不符合你的“品味”，也必须 100% 模仿它。不要触碰你修改目标之外的任何缩进或空行。

- **清理战场**：只删除因你的代码改动而变成“孤儿”的变量或导入。原有的冗余代码，只指出，不强删。

## 4. 契约与闭环执行

**无验证，不交付。将动作转化为可测试的契约。** 不要只是“完成任务”，要证明任务已完成：

- **定义 TDD 步骤**：在多步任务中，采用以下格式输出计划： `[ ] 步骤 1: 预期目标 -> 验证方式 (如：运行 xxx 测试)`。

- **处理隐性契约**：主动考虑该代码的边界条件（空值、并发、超时），并在 `<thinking>` 中简述如何处理了这些边界。
```

## 总结

Karpathy 风格 CLAUDE.md 的核心思想并不复杂：

```text
先思考
少设计
精准改
能验证
```

它并不会提升模型的能力上限，也无法让一个弱模型突然拥有更强的推理能力。

但在实际协作中，决定体验的往往不是模型能够做到什么，而是模型倾向于怎么做。

CLAUDE.md 的价值正在于此：它通过持续的预上下文注入，将团队认可的工程原则转化为 Agent 的行为偏好。

对于长期维护的软件项目而言，一个遵循共同工程原则的 Agent，往往比一个单纯能力更强的 Agent 更有价值。

如果你看到以下情况，说明这些指南正在发挥作用：

- diff 中不必要的改动更少 —— 只有请求的改动出现
- 因过度复杂而导致的重写更少 —— 代码第一次就写得简洁
- 澄清问题在实现之前提出 —— 而不是在犯错之后
- 干净、精简的 PR —— 没有顺带的重构或"改进"
