# 用 CLAUDE.md 约束 AI Agent

随着 Claude Code、Codex、Cursor 等 AI Agent 的普及，一个越来越常见的实践是在仓库根目录添加：

```text
CLAUDE.md
AGENTS.md
.cursor/rules/*
```

这类文件通常被称为项目指令（Project Instructions）或预上下文（Pre-context）。

它们的作用类似于：

```text
README.md -> 给开发者看
CLAUDE.md -> 给 AI 看
```

当 Agent 开始工作时，这些内容会被优先注入上下文，从而影响后续决策。

---

## 为什么需要预上下文

AI Agent 最大的问题并不是不会写代码，而是：

* 喜欢过度设计
* 喜欢顺手重构
* 喜欢引入依赖
* 喜欢猜测需求
* 喜欢修改无关代码

对于长期维护的项目来说，这些行为往往比实现功能本身更危险。

因此需要提前告诉 Agent：

```text
哪些事情应该做
哪些事情不应该做
```

以一个单一的、用于改善 Claude Code 的行为的 CLAUDE.md 文件为例，源自 Andrej Karpathy 的观察，关于 LLM 编码陷阱的总结。这个CLAUDE.md 正是为了解决上述问题而提出的。

---

## Karpathy 风格 CLAUDE.md

以下内容保持原样：

```md
## 1. 编码前思考

**不要盲目开工。强制暴露思考过程。**

- **使用 `<thinking>` 标签**：在代码块之前，简要列出你的理解、假设和权衡。

- **遇到歧义即阻断**：如果需求有多种解释，停止生成代码，列出选项 A 和 B，询问用户选择。

- **评估影响半径**：思考“这个改动会影响系统中的其他模块吗？”

## 2. 简洁优先

**代码是负债，越少越好。拒绝过度推测。**

- **标准库优先**：除非用户明确要求，否则禁止引入新的第三方依赖。

- **奥卡姆剃刀**：如果不需要复杂的泛型、接口或设计模式，就写最扁平、直白的函数。

- **无防备编程**：不要为了“未来可能的需求”或“极小概率的边缘情况”编写复杂的防错逻辑，除非这涉及到核心安全性或用户已明确提出。

## 3. 精准修改

**保持原有代码的纯洁性。禁止“顺手牵羊”。**

- **严禁使用 `...` 偷懒**：在提供修改后的代码时，必须提供**完整且可运行**的代码块，或者使用清晰的 Diff 格式。绝对不要用注释如 `// ... 现有代码保持不变 ...` 来糊弄用户，这会导致复制粘贴失败。

- **风格变色龙**：即使原代码的命名规范或格式化不符合你的“品味”，也必须 100% 模仿它。不要触碰你修改目标之外的任何缩进或空行。

- **清理战场**：只删除因你的代码改动而变成“孤儿”的变量或导入。原有的冗余代码，只指出，不强删。

## 4. 契约与闭环执行

**无验证，不交付。将动作转化为可测试的契约。** 不要只是“完成任务”，要证明任务已完成：

- **定义 TDD 步骤**：在多步任务中，采用以下格式输出计划： `[ ] 步骤 1: 预期目标 -> 验证方式 (如：运行 xxx 测试)`。

- **处理隐性契约**：主动考虑该代码的边界条件（空值、并发、超时），并在 `<thinking>` 中简述如何处理了这些边界。
```

## 这些规则为什么有效

### 1. 编码前思考

这是我认为收益最高的一条。

很多 Agent 的失败并非编码能力不足，而是对需求理解错误。

要求 Agent：

```text
先思考
再编码
```

能够显著降低返工概率。

特别是：

```text
遇到歧义即阻断
```

这实际上是在抑制模型最常见的行为：

```text
猜需求
```

---

### 2. 简洁优先

这一部分本质上是在对抗 Agent 的过度工程化倾向。

例如：

```text
用户需要一个函数
↓
Agent 创建三个抽象层
↓
引入两个依赖
↓
设计一个接口体系
```

Karpathy 的规则直接反向约束：

```text
先用标准库
先写最简单版本
不要为未来需求付费
```

这一原则在中小型项目中尤其有效。

---

### 3. 精准修改

这是团队协作价值最高的一条。

优秀的工程师往往具有：

```text
改什么
只改什么
```

的意识。

而 Agent 经常会：

```text
修一个 Bug
↓
顺手重构十个文件
```

因此：

```text
风格变色龙
禁止顺手修改
```

能够明显减少 Review 成本。

---

### 4. 契约与闭环执行

这一部分体现的是工程思维。

很多 Agent 停留在：

```text
代码写完
任务结束
```

而实际上：

```text
代码写完
↓
验证
↓
证明正确
↓
任务结束
```

才是完整闭环。

因此要求：

* 提供验证方式
* 提供测试步骤
* 考虑边界条件

是合理且必要的。

---

## 实验

为了验证这些规则是否有效，我基于同一项目进行了对照实验：

| 组别  | 配置                       |
| --- | ------------------------ |
| 对照组 | 默认 Agent                 |
| 实验组 | 注入 Karpathy 风格 CLAUDE.md |

测试任务包括：

* Bug 修复
* 新功能开发
* 重构任务
* 文档生成

观察指标：

* 修改文件数量
* 新增代码量
* 引入依赖数量
* 测试覆盖率
* 一次完成率

# 结果与分析

（实验数据待补充）

2. 是否减少新增依赖。
3. 是否提高一次完成率。
4. 是否降低代码复杂度。

如果这些指标得到改善，那么说明预上下文确实能够有效约束 Agent 的行为。

---

## 总结

Karpathy 风格 CLAUDE.md 的核心思想并不复杂：

```text
先思考
少设计
精准改
能验证
```

它并不会提升模型能力上限，但能够有效限制模型的不良工程习惯。

对于长期维护的软件项目而言，这类预上下文往往比更强的模型本身更重要。
