## ✍️ 投稿规范

为了保持整个站点的风格统一和极致的阅读体验，请务必遵守以下规范：

### 基本排版要求

- **中文排版**：使用中文全角标点（，。：；？！""），**中英文及数字之间建议保留一个空格**（例如：`VuePress 不错`）。
- **标题层级**：正文请从 `##`（二级标题）开始，避免全文出现单个 `<h1>` 或层级混乱。
- **文件命名**：一律使用**小写字母**，单词间使用连字符 `-` 连接。

| ✅ 正确示例 | ❌ 错误示例 |
| :--- | :--- |
| `stm32g0.md` | `STM32G0.md`、`stm32 g0.md` |
| `fpga-synthesis.md` | `FpgaSynthesis.md`、`fpga_synthesis.md` |

### 内容组件规范

请善用 VitePress 提供的内置容器组件来标识特殊内容，以提升视觉层级：

| 容器标签 | 使用场景 | 语法示例 |
| :--- | :--- | :--- |
| 💡 **tip** | 分享设计灵感、小技巧或补充信息 | `::: tip 标题` |
| ⚠️ **warning** | 提醒容易踩坑的注意事项，或可能导致的异常状况 | `::: warning 标题` |
| ⛔ **danger** | **必看！** 严重错误会导致硬件烧毁或数据丢失等不可逆转后果 | `::: danger 标题` |

**渲染效果与编写方式：**

```markdown
::: tip 设计小贴士
这里可以写一些你发现的巧妙电路连接方式。
:::

::: warning 注意事项
调试时注意区分 3.3V 和 5V 电平，避免烧毁芯片。
:::

::: danger 危险操作
接错电源正负极会直接烧毁晶振与芯片本体！
:::
```

### 代码块标准

所有代码块**必须**标注相应的语言类型，且**需包含必要的注释说明**，避免无意义的裸代码。

| 基本要求 | ✅ 正确示范 | ❌ 错误示范 |
| :--- | :--- | :--- |
| **显式声明语言** | ` ```c ` | ` ``` ` |
| **提供上下文** | 在代码块前/后作几句引言或总结 | 直接扔上一段代码就跑 |
| **注释规范** | `// 初始化系统时钟` 描述操作意图 | `// 1` 毫无意义的注释 |

**推荐示例：**

```c
// 优先采用清晰的注释，描述寄存器操作的目的
RCC->IOPENR |= RCC_IOPENR_GPIOAEN; // 开启 GPIOA 端口时钟
```

### 图表与公式

- **原理图/流程图**：简单的信号流或流程节点推荐使用 `Mermaid` 绘制，复杂的原理图请上传清晰、无过大水印的截图。
- **计算公式**：涉及数理逻辑（如电阻分压、截止频率）请一律采用 LaTeX 格式。

```markdown
电容截止频率：$f_c = \frac{1}{2\pi RC}$
电阻分压网络：$V_{out} = V_{in} \cdot \frac{R_2}{R_1 + R_2}$
```

> 渲染效果示例：$f_c = \frac{1}{2\pi RC}$

## 🎓 GitHub 新手教程

如果你从未使用过 GitHub，不用担心！本节将手把手教你完成从注册到提交的完整流程。

### 第一步：注册 GitHub 账号

1. 访问 [GitHub 官网](https://github.com)，点击右上角的 **Sign up**。
2. 依次输入邮箱、设置密码，并填写你的用户名。
   > 💡 **用户名建议**：尽量使用有意义的英文组合，如 `embedded-dev`、`stm32-fan` 等，避免使用纯随机乱码。
3. 进入邮箱查收验证码，完成验证。

### 第二步：Fork 仓库

**Fork** 的作用是把原仓库完全克隆一份到你自己的账号下，供你自由摸索测试，而不会干扰原库。

1. 进入本项目首页：[chengjp0825/doc_base_vitepress](https://github.com/chengjp0825/doc_base_vitepress)
2. 点击页面右上角的 **Fork** 按钮。
3. 在弹出的页面中确认信息并点击 **Create Fork**，稍等几秒即可跳转到你名下的专属仓库页面。

### 第三步：修改与编辑文件

我们提供两种修改方式，根据你的能力和需要选择：

#### 方案 A：直接网页端修改（零门槛，极速版）
适合修补错别字、修改几行文本等极小型改动。
1. 在你 Fork 出的仓库中，点进需要修改的文件。
2. 点击右上角的 **✏️ 铅笔图标 (Edit this file)** 即可进入编辑模式。
3. 完成修改后，滚动到页面最下方。在 "Commit message" 框内简要描述你的修改（如：\`修正 STM32 串口波特率笔误\`）。
4. 点击 **Commit changes**。

#### 方案 B：本地 Git 命令行（开发者进阶）
适用于编写长篇距，或者需要开本地服务器预览效果。

```bash
# 1. 下载你 Fork 的代码到本地（请将 YOUR_USERNAME 替换为自己的名字）
git clone https://github.com/YOUR_USERNAME/doc_base_vitepress.git

# 2. 进入克隆下来的文件夹
cd doc_base_vitepress

# 3. 创建并切换到一个新分支（分支名最好能概括你的改动内容）
git checkout -b add-stm32g0-notes

# 4. （此处使用你顺手的编辑器修改或增加文件内容）

# 5. 暂存所有改动
git add .

# 6. 提交你的改动并附加说明信息
git commit -m "docs: 新增 STM32G0 基础外设使用笔记"

# 7. 将修改推送到你的 GitHub 云端仓库
git push origin add-stm32g0-notes
```

### 第四步：提交 Pull Request (PR)

PR 是向原仓库发起合并请求的重要动作。

1. 打开你刚刚推送后的 GitHub 个人仓库主页。
2. 你会看到顶部浮现出一个醒目的横幅提示，点击绿色的 **Compare & pull request** 按钮。
3. 务必确认对比方向：从 `你的仓库/你的分支` 指向 `chengjp0825/doc_base_vitepress` 的 `main` 分支。
4. 填写 PR 详情：

   > 💡 **推荐的 PR 描述模板：**
   > ```markdown
   > ## 修改内容
   > - 新增了 I2C 时序讲解的章节
   > - 修正了原文档中的一处错别字
   >
   > ## 参考资料
   > - [I2C 官方规范](url)
   > ```
5. 确认无误后点击左下角的 **Create pull request**，随后只需耐心等待维护者的 Review 即可！ 🎉

### 补充技能：如何创建 Issue？

发现 Bug 但没时间修？有极好的点子想探讨？那就提 Issue 吧：

1. 返回本项目的[主仓库页面](https://github.com/chengjp0825/doc_base_vitepress)。
2. 点击上方的 **Issues** 选项卡。
3. 点击绿色的 **New issue** 按钮。
4. 根据你的意图选择相应的模版（Bug 报告 / 功能建议 / 提问 等）。
5. 尽量详细地说明情况，最后点击 **Submit new issue**。
